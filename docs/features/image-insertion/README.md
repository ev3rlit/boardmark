## 1. Overview

### 1.1 Problem Statement

현재 Boardmark는 텍스트, 노트, 프레임 중심의 캔버스 편집은 가능하지만 이미지 삽입이 불가능하다. 툴바에 `Image` 버튼이 존재하더라도 실제로는 비활성 상태이며, 사용자는 클립보드 이미지, 로컬 파일 이미지, 외부 URL 이미지를 캔버스에 올릴 수 없다.

이 제약 때문에 사용자는 참고 스크린샷, 다이어그램, 목업, 외부 시각 자료를 Boardmark 캔버스 안에서 직접 배치하지 못한다. 결과적으로 아이디어 보드, 기획 보드, UI 리뷰 보드 같은 주요 사용 시나리오가 제한된다.

### 1.2 Product Goal

Boardmark에서 이미지를 텍스트와 동일한 수준의 1급 캔버스 요소로 다룬다.

- 사용자는 붙여넣기, 툴바, 마크다운 문법으로 이미지를 삽입할 수 있어야 한다.
- 사용자는 드래그 앤 드롭으로도 이미지를 삽입할 수 있어야 한다.
- 로컬 이미지는 워크스페이스 안에 저장되고, 문서에는 상대 경로로 참조되어야 한다.
- 이미지는 캔버스 위에서 이동, 리사이즈, 비율 유지 조작이 가능해야 한다.
- 이미지 표현은 인라인 마크다운과 독립 `::: image` 블록을 모두 지원해야 한다.
- 삽입 직후부터 선택, 교체, 삭제, 복제, 실행 취소 같은 기본 편집 UX가 일관되게 동작해야 한다.
- 깨진 경로, 로딩 중 상태 같은 실패/대기 상태를 사용자에게 명확히 보여줘야 한다.
- 브라우저와 데스크톱에서 가능한 한 동일한 이미지 편집 경험을 제공해야 한다.

### 1.3 Success Criteria

- 신규 사용자가 별도 설명 없이 1분 내 이미지 하나를 캔버스에 추가할 수 있다.
- 붙여넣기와 파일 선택으로 삽입된 이미지는 문서 저장 후 다시 열어도 동일한 상대 경로로 복원된다.
- 로컬 이미지와 URL 이미지가 모두 렌더링되고, 기본 이동/리사이즈 조작이 안정적으로 동작한다.
- 이미지 삽입 후 별도 추가 조작 없이 바로 이동, 리사이즈, 삭제, 교체가 가능하다.
- 경로 오류나 로딩 지연이 발생해도 사용자는 현재 상태와 다음 액션을 이해할 수 있다.

## 2. Goals & Non-Goals

### Goals

- 클립보드 이미지 붙여넣기 지원
- 드래그 앤 드롭 이미지 삽입 지원
- 툴바 `Image` 버튼으로 이미지 추가 지원
- 이미지 추가 방식 2종 지원: 링크 방식, 파일 방식
- 마크다운 이미지 링크 렌더링 지원
- 신규 블록 컴포넌트 `::: image` 지원
- 이미지 비율 유지 필드 제공
- 이미지 리사이즈 지원
- 이미지 위치 이동 지원
- 이미지 교체, 삭제, 복제, Undo/Redo 연동 지원
- 멀티 선택 환경에서 이미지도 다른 오브젝트와 동일하게 동작
- 깨진 이미지 상태 및 로딩 상태 제공
- 삽입 직후 선택 상태, 초기 크기 규칙, 퀵 액션 제공
- 접근 가능한 `alt` 필드 지원
- 대용량 이미지의 최대 해상도 제한 적용

### Non-Goals

- 이미지 자르기, 회전, 필터, 보정
- 이미지 위 도형/주석 그리기
- 다중 이미지 일괄 업로드
- 외부 URL 이미지를 자동 다운로드해서 로컬 파일로 변환
- 이미지 레이어 잠금, 정렬, 그룹 편집
- 이미지 자르기 이후의 고급 편집 UI
- 이미지 주석 레이어 및 협업 코멘트
- 더블클릭 기반 고급 편집 모드

## 3. Users & Core Scenarios

### Target User

- 스크린샷과 레퍼런스를 섞어 캔버스를 구성하는 기획자
- 이미지와 텍스트를 같이 배치해 사고를 정리하는 개발자
- 로컬 파일 기반 워크플로우를 선호하는 AI-native 사용자

### Core User Stories

```text
AS  사용자
I WANT  클립보드의 이미지를 바로 붙여넣고
SO THAT 스크린샷을 빠르게 캔버스에 추가할 수 있다

AS  사용자
I WANT  툴바의 Image 버튼으로 로컬 파일 또는 링크를 선택해 이미지를 추가하고
SO THAT 붙여넣기 없이도 원하는 이미지를 삽입할 수 있다

AS  사용자
I WANT  로컬 이미지 파일을 캔버스로 드래그 앤 드롭해서 넣고
SO THAT 가장 빠른 방식으로 레퍼런스를 배치할 수 있다

AS  사용자
I WANT  마크다운 이미지 링크가 로컬 파일과 URL 모두 렌더링되고
SO THAT 기존 마크다운 자산을 그대로 활용할 수 있다

AS  사용자
I WANT  이미지를 독립된 `::: image` 블록으로 표현하고
SO THAT 위치, 크기, 비율을 캔버스 객체처럼 안정적으로 제어할 수 있다

AS  사용자
I WANT  이미지가 깨졌을 때 이유와 재설정 액션을 바로 보고
SO THAT 파일 이동이나 URL 오류를 빠르게 복구할 수 있다

AS  사용자
I WANT  이미지의 위치와 크기를 유지한 채 원본만 교체하고 실행 취소할 수 있고
SO THAT 편집 흐름을 끊지 않고 반복 수정할 수 있다
```

## 4. Product Rules

### 4.1 자산 수명 주기

- 이미지 오브젝트를 삭제해도 실제 에셋 파일은 삭제하지 않는다.
- 삭제는 문서 내 참조만 제거하는 동작으로 정의한다.

### 4.2 저장 전 문서 처리

- 새 문서가 아직 저장되지 않은 상태에서는 이미지 삽입 전에 먼저 문서 저장을 유도한다.
- 문서 저장 위치가 확정된 뒤 자산 저장과 참조 생성이 진행된다.

### 4.3 파일명 충돌 처리

- 자산 파일명 충돌 시 숫자 증가 suffix를 붙여 중복을 회피한다.
- 예시: `image.png`, `image-1.png`, `image-2.png`

### 4.4 상대 경로와 경로 복구

- 제품이 생성하는 로컬 이미지 참조는 상대 경로를 사용한다.
- 문서 이동, 폴더 이동, 다른 워크스페이스 복사 이후의 경로 복구는 제품 책임 범위에 포함하지 않는다.

### 4.5 마크다운 이미지와 `::: image`의 역할

- 마크다운 이미지와 `::: image`는 모두 유효한 이미지 표현이다.
- 어떤 표현을 쓸지는 사용자가 결정한다.
- 더 자유로운 캔버스 편집이 필요할 때 `::: image`를 사용하는 것을 권장한다.

## 5. Functional Requirements

### 5.1 삽입 방식 A: 클립보드 붙여넣기

- 마크다운 편집 상태가 아닐 때 클립보드에 이미지 데이터가 있으면 캔버스 이미지 삽입 플로우가 실행된다.
- 마크다운 편집 상태에서 클립보드에 이미지 데이터가 있으면 앱은 이미지를 로컬 파일로 저장하고 `![]()` 마크다운 링크를 자동 삽입한다.
- 새 문서가 아직 저장되지 않은 상태에서는 먼저 문서 저장을 유도한 뒤 이미지 추가를 진행한다.
- 앱은 클립보드 이미지 데이터를 로컬 파일로 생성한다.
- 생성된 파일은 현재 캔버스 문서 기준의 상대 경로로 참조한다.
- 캔버스 붙여넣기의 기본 결과는 `::: image` 블록 생성이다.

### 5.2 삽입 방식 B: 툴바 Image 버튼

- 플로팅 툴바의 `Image` 버튼을 활성화한다.
- 버튼 클릭 시 최소 2개 옵션을 제공한다.
- `Link`: 로컬 경로 또는 URL을 입력해서 참조형 이미지 추가
- `File`: 파일 선택기로 로컬 이미지를 선택해 삽입

#### Link 방식

- 사용자는 URL 또는 로컬 이미지 경로를 입력할 수 있다.
- 유효한 참조이면 이미지 노드를 생성한다.
- URL은 원문 그대로 유지한다.
- 수동 입력한 로컬 경로는 사용자가 입력한 경로를 유지하되, 문서 기준 상대 경로이면 그대로 저장한다.

#### File 방식

- 사용자가 로컬 파일을 선택하면 앱은 해당 파일을 문서 자산 위치로 복사한 뒤 `::: image` 블록을 생성한다.
- 저장되는 `src`는 현재 문서 기준 상대 경로여야 한다.
- 새 문서가 아직 저장되지 않은 상태에서는 먼저 문서 저장을 유도한 뒤 이미지 추가를 진행한다.

### 5.3 삽입 방식 C: 드래그 앤 드롭

- 사용자는 로컬 이미지 파일을 캔버스 위로 드래그 앤 드롭할 수 있다.
- 드롭은 항상 캔버스에 독립 이미지 객체를 생성한다.
- 드롭 시 앱은 `File` 삽입 방식과 동일한 저장 규칙을 적용한다.
- 드롭 위치가 유효하면 해당 지점 또는 가장 가까운 캔버스 좌표에 이미지가 생성된다.
- 드롭 직후 이미지는 선택 상태가 되며 바로 이동/리사이즈 가능해야 한다.
- 새 문서가 아직 저장되지 않은 상태에서는 먼저 문서 저장을 유도한 뒤 이미지 추가를 진행한다.

### 5.4 삽입 방식 D: 마크다운 이미지 링크

- 노트 본문 또는 마크다운 렌더링 영역에서 표준 이미지 문법을 지원한다.

```md
![alt text](./assets/mockup.png)
![alt text](https://example.com/mockup.png)
```

- 로컬 파일 경로와 URL 경로를 모두 렌더링해야 한다.
- 시스템이 자동 생성하는 로컬 이미지 참조는 상대 경로를 기본값으로 사용한다.
- 마크다운 이미지 렌더링은 기존 마크다운 콘텐츠 흐름을 깨지 않아야 한다.

### 5.5 신규 블록 `::: image`

- 이미지 전용 블록 타입 `image`를 추가한다.
- `::: image`는 캔버스 상의 독립 객체로 취급한다.
- 이번 버전의 `::: image`는 본문 없는 단일 헤더 문법으로 고정한다.
- 캡션은 본문이 아니라 선택적 메타데이터 필드 `title`로만 표현한다.

확정 문법:

```md
::: image { id: hero-shot, src: "./board.assets/hero-shot.png", alt: "메인 화면 캡처", title: "메인 화면", lockAspectRatio: true, at: { x: 240, y: 160, w: 420, h: 280 } }
:::
```

필드 규칙:

- `id`: 문서 내 고유 식별자
- `src`: 상대 로컬 경로 또는 URL
- `alt`: 접근성 및 문서 가독성을 위한 대체 텍스트
- `title`: 선택적 표시용 제목
- `lockAspectRatio`: 비율 유지 여부
- `at.x`, `at.y`: 캔버스 위치
- `at.w`, `at.h`: 캔버스 크기

### 5.6 편집 액션

- 이미지 객체는 최소한 아래 액션을 지원해야 한다.
- `Replace image`: 위치와 크기를 유지한 채 원본만 교체
- `Delete`: 문서에서 이미지 객체 제거
- `Duplicate`: 동일 속성의 이미지 객체 복제
- `Copy / Paste`: 선택된 이미지 객체를 복사하고 재삽입
- 이미지 객체 복제와 복사-붙여넣기는 동일 자산 파일의 `src`를 재사용하며 새 파일을 만들지 않는다.
- 이미지 객체 삭제 시 실제 에셋 파일은 삭제하지 않고 유지한다.
- 위 액션은 Undo/Redo 히스토리에 통합되어야 한다.

## 6. Interaction Requirements

### 6.1 비율 유지 필드

- 이미지 선택 시 비율 유지 필드를 노출한다.
- 기본값은 `켜짐`으로 한다.
- 비율 유지가 켜져 있으면 리사이즈 시 원본 종횡비를 유지한다.
- 비율 유지가 꺼져 있으면 자유 리사이즈를 허용한다.

### 6.2 초기 배치와 선택 상태

- 삽입 직후 이미지는 항상 선택 상태가 된다.
- 기본 삽입 위치는 다음 우선순위를 따른다.
- 마지막 포인터 위치가 있으면 해당 위치
- 그렇지 않으면 현재 뷰포트 중심
- 선택된 프레임 또는 그룹 내부 삽입 규칙이 이후 추가되면 그 규칙이 우선한다.
- 초기 크기는 원본 크기를 읽어 설정하되, 화면을 과도하게 가리지 않도록 안전한 최대 표시 크기를 둔다.
- 대용량 이미지는 원본을 그대로 저장하지 않고 비율을 유지한 채 최대 해상도 제한 안으로 축소해 저장한다.

### 6.3 리사이즈

- 사용자는 선택된 이미지의 핸들을 드래그해 크기를 변경할 수 있다.
- 조작 결과는 문서의 `at.w`, `at.h`에 반영된다.
- 비율 유지가 켜져 있으면 `w` 또는 `h` 한 축 변경에 맞춰 다른 축이 자동 계산된다.
- 최소 크기 제한을 두어 조작 불가능한 크기로 축소되지 않게 한다.

### 6.4 위치 이동

- 사용자는 이미지를 드래그해 캔버스 상 위치를 변경할 수 있다.
- 조작 결과는 문서의 `at.x`, `at.y`에 반영된다.
- 이동 중 현재 좌표 또는 정렬 가이드를 표시해 배치를 돕는다.

### 6.5 선택 시 퀵 액션

- 이미지 선택 시 컨텍스트 패널 또는 퀵 액션에서 아래 항목을 제공한다.
- `Replace image`
- `Open source`
- `Reveal file` 또는 로컬 파일 위치 열기
- `Reset size`
- `Lock aspect ratio`
- `Alt text` 편집
- 더블클릭은 이번 기능 범위에서 아무 동작도 하지 않는다.

### 6.6 로딩 및 오류 상태

- 이미지 로딩 중에는 빈 영역 대신 플레이스홀더 또는 스켈레톤을 표시한다.
- 로컬 파일 누락 시 이미지 썸네일에 파일을 찾을 수 없다는 상태를 표시한다.
- 권한 오류가 발생하면 명시적 에러 메시지를 보여준다.
- URL 실패 시 깨진 이미지 상태를 명시적으로 보여준다.
- 오류 상태에서는 최소한 `경로 보기`와 `이미지 교체` 액션을 제공한다.
- 실패를 조용히 무시하거나 아무 표시 없이 숨기지 않는다.

### 6.7 키보드 및 편집 일관성

- `Delete` 또는 `Backspace`로 선택된 이미지 삭제
- `Cmd/Ctrl+D`로 복제
- 화살표 키로 미세 이동
- Undo/Redo 단축키와 완전 연동
- 노트, 프레임 등 기존 객체와 최대한 동일한 조작 모델 유지
- 멀티 선택은 캔버스의 공통 선택 모델을 그대로 따른다.
- 이미지 역시 멀티 선택 이동, 삭제, 복제 대상에 포함된다.

### 6.8 붙여넣기와 드롭 의도 판별

- 마크다운 편집 상태에서의 이미지 붙여넣기는 `![]()` 마크다운 삽입으로 처리한다.
- 캔버스 편집 상태에서의 이미지 붙여넣기는 독립 `::: image` 생성으로 처리한다.
- 드롭은 항상 캔버스에 독립 이미지 객체를 배치한다.

## 7. Content, Storage, and Platform Rules

### 7.1 로컬 자산 저장 규칙

- 붙여넣기 또는 파일 방식으로 추가된 이미지는 현재 문서 옆의 전용 자산 디렉터리에 저장한다.
- 자산 디렉터리 이름은 문서와 연관성을 알 수 있어야 한다.
- 예시: `untitled.canvas.md`를 편집 중이면 `untitled.assets/` 또는 `untitled.canvas.assets/` 아래에 이미지 저장
- 문서 내부의 `src`는 항상 자산 디렉터리를 기준으로 한 상대 경로를 사용한다.

예시:

```md
src: ./untitled.canvas.assets/pasted-image-001.png
src: ./untitled.canvas.assets/pasted-image-002.png
```

### 7.2 경로 해석 규칙

- `::: image`의 `src`는 URL 또는 로컬 경로를 허용한다.
- 상대 로컬 경로는 현재 `.canvas.md` 파일 위치를 기준으로 해석한다.
- URL은 원격 자원으로 처리한다.
- 제품이 생성하는 로컬 경로는 상대 경로를 우선한다.
- 경로 복구는 별도 수행하지 않는다.

### 7.3 포맷 정책

- 제품 목표는 가능한 한 모든 이미지 포맷을 폭넓게 지원하는 것이다.
- 붙여넣기, 파일 삽입, 드롭 삽입은 현재 런타임에서 읽고 디코딩 가능한 모든 이미지 포맷을 대상으로 시도한다.
- 형식 미지원이 확인되면 붙여넣기 또는 삽입을 무시한다.

### 7.4 EXIF 정책

- 파일 삽입, 드롭 삽입, 클립보드 삽입으로 새로 저장되는 로컬 자산은 EXIF orientation을 반영한 정방향 이미지로 정규화한다.
- 이를 통해 웹과 데스크톱에서 회전 결과가 다르게 보이는 문제를 줄인다.
- 링크 방식으로 참조된 원격 URL 또는 기존 로컬 파일은 원본을 재작성하지 않는다.

### 7.5 원격 URL 보안 정책

- 원격 URL 이미지는 가능한 한 `https`를 우선 사용한다.
- 브라우저와 데스크톱 모두 보안 정책을 우회하는 별도 프록시, 비밀 헤더 주입, 무단 다운로드 저장을 수행하지 않는다.
- 원격 URL 실패는 명시적 오류 상태로 표시한다.

### 7.6 플랫폼 일관성 정책

- 브라우저와 데스크톱은 가능한 한 동일한 사용자 경험과 문법을 제공해야 한다.
- 다만 플랫폼 보안 제약으로 인해 로컬 파일 접근 방식이 다를 경우 동일한 사용자 목적을 달성하는 범위에서 가장 가까운 UX를 제공한다.

## 8. UX Flow

### 8.1 붙여넣기 플로우

1. 사용자가 OS 캡처 도구 등으로 이미지를 클립보드에 복사한다.
2. 사용자가 현재 상태가 마크다운 편집인지 캔버스 편집인지에 따라 붙여넣기를 실행한다.
3. 문서가 저장되지 않았다면 앱이 먼저 저장을 유도한다.
4. 마크다운 편집 상태이면 앱이 로컬 파일을 저장하고 `![]()` 링크를 삽입한다.
5. 캔버스 편집 상태이면 앱이 로컬 파일 생성 후 `::: image` 블록을 문서에 삽입한다.
6. 캔버스 삽입 결과는 즉시 표시되고 선택 상태가 된다.

### 8.2 툴바 플로우

1. 사용자가 플로팅 툴바에서 `Image` 버튼을 누른다.
2. `Link` 또는 `File` 방식을 선택한다.
3. 입력 또는 파일 선택을 완료한다.
4. 앱이 이미지 노드를 생성하고 캔버스에 배치한다.

### 8.3 드래그 앤 드롭 플로우

1. 사용자가 로컬 이미지 파일을 캔버스로 드래그한다.
2. 유효한 드롭 영역 위에서 드롭한다.
3. 문서가 저장되지 않았다면 앱이 먼저 저장을 유도한다.
4. 앱이 문서 자산 디렉터리로 파일을 복사하고 이미지 노드를 생성한다.
5. 이미지는 드롭 위치에 배치되고 선택 상태가 된다.

### 8.4 마크다운 플로우

1. 사용자가 노트 본문 또는 문서에 표준 마크다운 이미지 링크를 작성한다.
2. 렌더러가 로컬 경로 또는 URL을 해석한다.
3. 이미지가 문서 렌더링 결과에 포함된다.

### 8.5 오류 복구 플로우

1. 사용자가 이미지가 깨진 상태를 본다.
2. 앱은 파일 누락 또는 로딩 실패 상태를 명확히 표시한다.
3. 사용자는 `Replace image` 또는 경로 확인 액션을 선택한다.
4. 복구 후 동일한 위치와 크기로 이미지가 정상 표시된다.

## 9. Acceptance Criteria

### Functional

- 마크다운 편집 상태의 클립보드 붙여넣기는 로컬 파일 저장 후 `![]()` 링크를 삽입한다.
- 캔버스 편집 상태의 클립보드 붙여넣기는 로컬 파일 생성 후 `::: image` 블록을 생성한다.
- 로컬 이미지 파일을 드래그 앤 드롭으로 삽입할 수 있다.
- 툴바 `Image` 버튼이 더 이상 비활성 상태가 아니며 `Link`, `File` 두 방식으로 동작한다.
- 마크다운 이미지 링크가 로컬 파일과 URL 모두 렌더링된다.
- `::: image` 블록을 파서와 렌더러가 인식한다.
- 이미지 선택 후 비율 유지 필드를 조작할 수 있다.
- 이미지 이동과 리사이즈 결과가 문서에 반영되고 재오픈 후 유지된다.
- 이미지 교체, 삭제, 복제, Undo/Redo가 정상 동작한다.
- `alt` 필드를 생성/수정할 수 있다.
- 새 문서가 저장되지 않았다면 이미지 삽입 전에 저장이 선행된다.
- 파일명 충돌 시 숫자 증가 규칙으로 새 파일명이 생성된다.
- 이미지 복제와 복사-붙여넣기는 동일 자산 `src`를 재사용한다.
- 이미지 삭제 시 실제 에셋 파일은 유지된다.
- `::: image` 문법은 본문 없는 단일 헤더 구조로 동작한다.

### UX

- 붙여넣기 후 1초 이내에 이미지가 캔버스에 표시된다.
- 드롭 또는 파일 선택 직후 이미지가 선택 상태로 표시된다.
- 기본 상태에서 사용자는 별도 설정 없이 이미지를 이동하고 리사이즈할 수 있다.
- 비율 유지가 켜진 상태에서 리사이즈 시 이미지가 찌그러지지 않는다.
- 로딩 중 상태와 깨진 경로 상태가 시각적으로 구분된다.
- 파일 누락 시 썸네일 영역에 못 찾는다는 메시지가 표시된다.
- 권한 오류 시 명시적 에러 메시지가 표시된다.
- 오류 상태에서 사용자는 이미지 교체 또는 경로 확인 액션을 찾을 수 있다.
- 선택 상태에서 핵심 액션이 1회 클릭 이내로 접근 가능하다.
- 더블클릭은 아무 동작도 하지 않는다.
- 멀티 선택 환경에서 이미지도 다른 오브젝트와 동일한 편집 경험을 가진다.

### Persistence

- 저장 후 문서를 다시 열어도 로컬 이미지 경로가 깨지지 않는다.
- 상대 경로 기반 로컬 이미지는 현재 문서 위치 기준에서만 해석하며 경로 복구는 별도 수행하지 않는다.

## 10. Constraints & Risks

- 브라우저와 데스크톱은 최대한 동일하게 지원하되, 로컬 파일 접근과 보안 제약은 구현 난이도와 검증 범위를 높인다.
- 모든 포맷을 폭넓게 지원하는 목표는 플랫폼별 디코더 차이와 포맷별 메타데이터 차이 때문에 테스트 비용이 크다.
- 큰 이미지가 많은 보드에서는 디코딩 비용과 메모리 사용량이 급증하므로 import 단계 해상도 제한과 렌더 단계 최적화가 필요하다.
- 원격 URL 이미지는 네트워크, CSP, mixed content, 추적성 이슈를 동반하므로 실패 상태와 보안 정책을 명확히 유지해야 한다.

---

## 11. Open Decisions

- 자산 디렉터리 이름을 `*.assets`로 할지 `*.canvas.assets`로 할지 최종 확정 필요
- 마크다운 이미지의 클릭 확대나 원본 열기 같은 부가 상호작용은 후속 범위로 분리할지 결정 필요
- 정렬 스냅과 가이드 표시를 이미지 MVP에 포함할지 전체 캔버스 공통 기능로 분리할지 결정 필요
- 최대 해상도 제한의 기본값을 얼마로 둘지 최종 확정 필요

## 12. Suggested Delivery Scope

### Phase 1

- `::: image` 고정 문법 확정 및 파서 반영
- 이미지 렌더러 추가
- 이동 / 리사이즈 / 비율 유지 지원
- 초기 배치 규칙 / 선택 상태 / 로딩 및 오류 상태 지원
- 삭제 / 복제 / Replace / Undo-Redo 연동
- 멀티 선택 연동

### Phase 2

- 클립보드 붙여넣기 지원
- 툴바 `Image > File` 지원
- 드래그 앤 드롭 지원
- 로컬 자산 저장 및 상대 경로 생성
- 저장 전 문서 처리, 파일명 충돌 회피, EXIF 정규화
- 대용량 이미지 최대 해상도 제한 적용

### Phase 3

- 툴바 `Image > Link` 지원
- 마크다운 이미지 링크의 로컬/URL 렌더링 정리
- 원격 URL 보안 정책 정교화
- 성능 최적화, 문서/파서/테스트 보강
